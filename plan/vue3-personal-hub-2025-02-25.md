# Vue3 + TS + Vite 个人中心 / 作品集 可实施计划

**创建时间**: 2025-02-25  
**目标**: 纯前端、可扩展，部署 GitHub Pages；包含学习收藏、小工具、后续项目展示。  
**说明**: 项目已迁移至仓库/目录 `personal-blog`，部署时 `base` 与仓库名一致（见 `vite.config.ts`）。

---

## 一、角色视角汇总

### 1. 产品视角 (PM)

| 维度 | 内容 |
|------|------|
| **用户画像** | 你自己（主用户）+ 访客（快速了解你、使用小工具） |
| **核心场景** | ① 收藏/回顾学习文章与链接 ② 日常使用小工具（图片压缩/转换等） ③ 展示个人项目与能力 |
| **功能优先级** | **P0**: 导航与首页、学习收藏（含分类/文件夹、导入导出、浏览器书签导入）、至少 1 个小工具（如图片压缩）<br>**P1**: 小工具集（见下方工具清单）、项目展示区、搜索/筛选、中英文切换、主题明暗切换<br>**P2**: 统计、更多扩展 |
| **关键依赖** | 纯前端 → 数据持久化可选：localStorage / IndexedDB / 后续可接 GitHub API 或静态 JSON |

### 2. 架构视角 (CTO)

| 维度 | 建议 |
|------|------|
| **整体形态** | 单页应用 (SPA)，按「模块」可插拔扩展（学习收藏 / 工具集 / 项目展示 各为一类模块） |
| **技术选型** | Vue 3 + TypeScript + Vite；路由 Vue Router；状态按需（Pinia 或 组合式 + provide/inject）；样式：CSS 变量 + 可选 UnoCSS/Tailwind |
| **可扩展点** | ① 路由与菜单由配置驱动 ② 小工具以「工具卡片 + 独立页面」形式注册 ③ 项目展示用静态配置（JSON/ frontmatter）即可 |
| **GitHub Pages** | 使用 `vite.config` 的 `base: '/repo-name/'`；构建产物 `dist` 部署到 `gh-pages` 分支或 GitHub Actions 自动部署 |

### 3. 前端视角 (Frontend)

| 维度 | 建议 |
|------|------|
| **路由设计** | `/` 首页，`/bookmarks` 学习收藏，`/tools` 工具列表，`/tools/:toolId` 具体工具，`/projects` 项目展示，`/projects/:id` 项目详情 |
| **数据与 API** | 无后端：收藏数据存 localStorage/IndexedDB；工具纯前端计算；项目数据用静态 JSON 或 MD 解析（如 vite-plugin-md）。 |
| **性能与体验** | 路由懒加载；工具页按需加载；图片压缩等用 Web Worker 避免卡顿；首屏只加载首页 + 当前路由组件。 |

## 二、可实施开发计划（分阶段）

### 阶段 0：项目初始化（1 天内）

1. 使用官方模板创建项目：`npm create vite@latest <project-name> -- --template vue-ts`
2. 配置：`base` 适配 GitHub Pages、路径别名 `@/`、TS 严格模式、ESLint + Prettier
3. 目录结构草案（见下方「推荐目录结构」）
4. 实现一个简单首页（标题 + 导航入口），部署到 GitHub Pages 跑通

### 阶段 1：骨架与学习收藏（MVP 核心）

1. **布局与导航**
   - 顶部/侧边导航，响应式；路由：`/`、`/bookmarks`、`/tools`、`/projects`（可先占位）
2. **学习收藏模块**（已确认需求）
   - 数据模型：`id, title, url, type(article|video|other), tags[], folderId?, createdAt`；**文件夹**：`id, name, order`
   - **分类/文件夹**：支持创建文件夹、收藏归属到文件夹、列表按文件夹分组或筛选
   - 列表页：展示、按文件夹/标签/类型筛选、搜索（标题/URL）
   - 新增/编辑表单（弹窗或独立页）；删除
   - **导入/导出**：导出为 JSON 或 HTML 书签格式；支持从 JSON/HTML 文件导入
   - **从浏览器书签导入**：支持上传浏览器导出的书签 HTML（Chrome/Edge 等），解析后批量导入并可选择文件夹
   - 持久化：localStorage 或 IndexedDB（推荐 IndexedDB 便于后续扩展）
3. **工具列表页**
   - `/tools` 展示「工具卡片」列表（图标、名称、简短描述、跳转链接）
   - 至少实现 **1 个小工具**：例如「图片压缩」（选择图片 → 调整质量/尺寸 → 下载），作为后续工具的模板

**验收要点**：能添加/编辑/删除收藏并持久化；支持文件夹与导入/导出/浏览器书签导入；能打开工具列表并完成一次图片压缩流程。

### 阶段 2：小工具扩展与项目展示

1. **小工具扩展**（已确认工具清单）
   - 定义「工具注册接口」：`id, name, description, path, icon`
   - **工具清单**（均纯前端实现）：
     - 图片：图片压缩、图片格式转换（如 WebP ↔ PNG）
     - 文本/数据：**JSON 格式化**、**Base64 编解码**
     - 转换：**颜色转换**（如 HEX ↔ RGB ↔ HSL）、**时间戳转换**（时间戳 ↔ 日期时间、多时区等）
   - 以上工具通过配置注册，按需懒加载
2. **项目展示**（待定：见下方「面试官视角」分析后再与产品讨论确定）
   - 数据源：**仓库内 JSON/MD**（构建时或运行时读取，不影响 GitHub Pages 部署）
   - 数据：`projects.json` 或 `projects/*.md`，字段待与「项目展示需求」一起定稿
   - `/projects` 卡片列表，`/projects/:id` 详情页（可渲染 MD 描述）
   - 若用 MD：可配合 `vite-plugin-md` 或构建时生成 JSON，保证部署兼容性

**验收要点**：上述小工具均通过配置出现且可用；项目展示结构与字段在需求确定后实现并验收。

### 阶段 3：体验与可扩展性收尾

1. **风格与国际化**（已确认需求）
   - **UI/产品先出一版**：由 UI 与产品给出首版视觉与信息结构方案，供你确认后再开发落地
   - **中英文切换**：i18n（如 vue-i18n），语言持久化（localStorage + URL 参数可选）
   - **主题明暗切换**：亮色/暗色主题，CSS 变量 + 持久化偏好，可选跟随系统
2. 加载态、空状态、错误态（如工具加载失败、存储失败提示）
3. 简单 SEO：标题与 meta 按路由更新（Vue Router + document.title）
4. 可访问性：焦点管理、键盘导航、aria 属性
5. 文档：README 中说明如何「新增一个工具」「新增一个项目」（可扩展性说明）

## 三、项目展示：面试官/招聘方希望看到什么（供讨论确定需求）

在确定项目展示的字段、筛选、双入口等前，先做需求侧分析，便于产品与你自己拍板。

| 面试官关注点 | 说明 | 对本站的建议 |
|--------------|------|--------------|
| **能否快速判断技术栈与角色** | 希望一眼看到「用了什么技术、你在其中负责什么」 | 每个项目明确标注：技术栈（如 Vue3/TS/Vite）、你的角色（全栈/前端/主导）、项目类型（个人/实习/开源） |
| **是否有可运行的成果** | 更愿意点开「能跑」的 demo，而不是只有描述 | 提供 **在线 demo 链接**（如 GitHub Pages / Vercel）；若无法在线则写清原因（如需本地环境） |
| **源码是否可审阅** | 会通过代码质量、结构、注释判断能力 | 提供 **源码链接**（GitHub 等），且仓库 README、结构清晰 |
| **问题与亮点** | 想听「难点与解决」「亮点与取舍」 | 详情页或 MD 中可留「难点与解决」「技术亮点」简短段落，便于面试展开 |
| **时间与真实性** | 避免「堆积项目名」、希望有时间线 | 展示 **时间/周期**（如 2024.Q3）；若为学习/仿写可标注「练习项目」以保真实 |
| **与岗位的匹配度** | 前端岗会重点看前端项目、工程化、性能等 | 支持按 **技术栈/类型筛选**（如「Vue」「纯前端」「带 Node」），便于访客按需查看 |

**建议结论（待你与产品确认）**  
- 项目展示至少包含：**名称、简短描述、技术栈、你的角色、在线 demo 链接、源码链接、时间**；可选：封面图、难点与亮点、项目类型标签。  
- 列表支持按技术栈/类型筛选；详情页可渲染 MD，便于写「难点与解决」「技术亮点」。  
- 数据源采用 **仓库内 JSON/MD**（见下方「数据与部署」），在不影响 GitHub 部署的前提下使用。

## 四、数据源与部署（已确认）

- **采用仓库内 JSON/MD 作为数据源**：项目数据、静态文案等可放在仓库（如 `data/projects.json`、`content/projects/*.md`），构建时或运行时读取。
- **不影响 GitHub 正常部署**：构建阶段将 JSON/MD 打进 bundle 或生成静态路由即可，不依赖服务端；GitHub Pages 仍只托管静态产物，部署方式不变。
- 学习收藏等个人数据仍为浏览器端存储（localStorage/IndexedDB），与仓库数据源分离。

## 七、风险与注意点

| 风险 | 等级 | 应对 |
|------|------|------|
| GitHub Pages 刷新 404 | 中 | 使用 Hash 路由 `createWebHashHistory()` 或 404 重定向到 index.html（需服务器配置，Pages 默认支持部分方式） |
| 存储容量与迁移 | 低 | 先用 localStorage，数据量增大再迁 IndexedDB；如需多端同步可后续接简单后端或 GitHub Gist |
| 小工具依赖体积 | 低 | 按路由懒加载；大库（如 heavy 图片处理）放该路由 chunk，避免进主包 |
